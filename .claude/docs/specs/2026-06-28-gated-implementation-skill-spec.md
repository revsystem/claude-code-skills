# ゲート付き実装スキル 設計仕様

日時: 2026-06-28

## 目的・コンセプト

research-plan-annotate が生成した計画（plan.md のユニット・グルーピング）を入力に、実装をユニット単位のゲートで区切りながら進める新スキルを作る。各ユニットで「実装 → 検証 → 人によるレビュー・承認 → 次ユニット」を繰り返す。

狙いは human-in-the-loop の実装フェーズ版を確立すること。現状 research-plan-annotate は計画フェーズにしか承認ゲートを持たず、Handoff 後の実装はノーゲートで最後まで走る設計になっている（「すべてのタスクとフェーズが完了するまで停止しないでください」）。長く走るほど誤った方向に進み続けるリスクが高く、最後にまとめてレビューすると修正コストが跳ね上がる。AI-DLC（AWS Labs の AI コーディングエージェント向け steering rules 集。Inception→Construction→Operations の3フェーズ適応型）の理想であるユニット境界でのゲートを実装フェーズに持ち込み、誤りを早期に捕捉する。

## 想定利用者と利用シーン

利用者は research-plan-annotate で計画を固めた後、実装に移る開発者。

主な利用シーン:

- plan.md が完成し、ユニット・グルーピング済みのタスクリストを実装する
- 中断したセッションの実装を、未完了ユニットから再開する
- 計画スキルを通さない小規模作業を、最小計画を挟んでゲート付きで実装する

最重要前提: このスキルは実装フェーズを担うため「don't implement yet」境界を意図的に越える。ただし規律は形を変えて引き継がれ、核心原則は「ゲートを越えて次のユニットに進まない」。承認なしに走り続けることを禁じる、計画スキルの精神の実装フェーズ版とする。

## スコープ

含めること:

- plan.md のユニット・グルーピングをゲート境界として消費する実装ループ
- ユニットごとの検証（テスト・ビルドを実際に実行し、証拠を添えてレビューに出す）
- リスクと深度に応じたゲート種別の自動選択
- plan.md への完了マーク更新と、中断セッションの再開検知
- 全ユニット完了後の result.md 記録（`.claude/docs/reports/`）
- plan.md が無い場合の最小計画フォールバック経路
- research-plan-annotate 側 Phase 5 Handoff の文言差し替え

含めないこと:

- 計画・調査・設計フェーズ（research-plan-annotate の責務として維持）
- result.md の作成テンプレート新設（既存 assets/result-template.md を流用）
- 複数プロジェクト横断の成果物参照（スキルは現在の作業ディレクトリの `.claude/docs/` のみを見る）

線引きの根拠: 計画スキルと実装スキルを対称的な単一責任に分ける（アーキテクチャ設計で採用したスキル分割案）。計画と実行を分離することで各スキルの責任が明確になり、実装ゲートの強制力も高まる。スコープは AI-DLC の Construction（実装・テスト）から Operations 入口（結果記録）まで一気通貫でカバーし、2本のスキルで Inception→Construction→Operations の流れがほぼ閉じる。

## アーキテクチャ設計

### 検討した代替案とスキル分割の判断

実装フェーズのゲートをどこに置くかで3案を比較した。

- 案 A: research-plan-annotate を拡張する。Handoff にユニットゲートの指示を追加するだけ。利点: 導入コストが低い。欠点: 実装セッション中の強制力が弱く、指示を忘れると素通りする
- 案 B: 実装フェーズ専用スキルを新設する（本仕様で採用）。research-plan-annotate は計画まで、新スキルは実行と管理に専念する単一責任の分離。利点: 各スキルの責任が明確、実装ゲートの強制力が高い、ユニット構造をそのまま入力に受け取れる。欠点: スキルが増え呼び分けの手間が生じる
- 案 C: 「計画→実装→レビュー」を一貫して扱う統合スキルに再設計する。利点: Inception→Construction を一気通貫でカバー。欠点: スキルが肥大化し維持コストが上がる

案 B を採用する。計画と実行を分けることで各スキルの責任が明確になり、実装ゲートの強制力も高まる。research-plan-annotate が生成するタスクリストのユニット構造をそのまま入力として受け取れるため、つなぎも自然。

### 全体構成

2スキル構成。research-plan-annotate（計画、Inception）が plan.md を生成し、新スキル（実装、Construction→Operations 入口）がそれを消費する。

接合面は plan.md のユニット・グルーピング。既存のユニット概念は「分割の単位」だけでなく「承認ゲートの単位」という意味を後から獲得する。新たに作り直す必要はなく、出力をそのまま入力として受け渡す。

invocation は research-plan-annotate と同じく `disable-model-invocation: true`。実装は人が明示的に開始する工程であるべきで、勝手な実装着手を防ぐ。

### 着手前の解決フロー（ゼロ番目のゲート）

1. スキルは現在の作業ディレクトリの `.claude/docs/plans/` のみを参照する
2. 引数で対象を指定された場合（例: `/<skill> プロジェクトAの実装を開始して`）、その語をファイル名・トピックslugに照合する
   - 一意に決まれば、plan.md とヘッダがリンクする spec.md / research.md を読み、「この計画を実装します」と提示して確認を取る
   - 曖昧で複数候補に当たる場合は一覧から選ばせる
3. 引数なしの場合、候補 plan.md を日付の新しい順に、トピック名・日付・完了状況を添えて一覧提示し、選ばせる。最新を黙って自動選択しない（誤った計画の実装は巻き戻しの利かない高コストの失敗のため）
4. 選んだ plan.md のユニットに完了マークがある場合、中断セッションと判断し「次の未完了ユニットから再開／最初からやり直し」を確認する

### ユニットごとの実装ループ

実装 → 検証（テスト・ビルドを実行）→ ゲート（検証結果を添えて提示）→ 承認 → plan.md に完了マーク → 次ユニット。

検証を各ユニット境界で走らせるのが核心。人は「やったと主張するコード」ではなく「動く証拠付きのコード」をレビューする。superpowers の verification-before-completion（evidence before assertions）と揃える。

## 技術選定と根拠

実装言語はスキル（Markdown の SKILL.md）。既存スキル（research-plan-annotate / handover）と同じ Claude Code スキル形式に従う。

result.md テンプレートは新設せず research-plan-annotate の assets/result-template.md を流用する。reports は種類が多様で「参考の出発点」として既存テンプレートが既にその前提で書かれているため、重複を作らない。

ゲート選択ロジックは SKILL.md 内にインラインで記述する（参照ファイルを新設しない）。前回の AI-DLC 統合では、選択式フォーマットの提示方法として参照ファイル新設とテンプレート直書きを比較し、シンプルさ優先でテンプレート直書きを選んだ。その判断と整合させ、本スキルでもロジックを別ファイルに切り出さずインラインに保つ。

## データ設計

主要アーティファクトと所在:

- 入力: plan.md（`.claude/docs/plans/YYYY-MM-DD-<feature>-plan.md`）、リンク先の spec.md / research.md
- 更新: plan.md のユニット完了マーク（実装の進行に伴い更新）
- 出力: result.md（`.claude/docs/reports/YYYY-MM-DD-<topic>-result.md`）

result.md は本来 plan.md / research.md へリンクする。単体実行ではこの上流リンクが空になる。これは欠陥ではなく単体経路の自然な帰結として扱い、result.md にも上流アーティファクトが無い旨を残す。

## 要件ごとの実装方針

ゲート種別の自動選択（リスクと深度に応じた使い分け）:

3種のゲートを使い分ける。

- 事前宣言型（着手前に何をするか宣言し承認を取ってから着手）
- diff レビュー型（完了後に diff を見せて承認を取る）
- 進捗確認型（完了後に何をやったか要約報告し続行可否だけ確認）

リスク判定の材料: 不可逆性（マイグレーション・データ削除・外部API書き込み・デプロイ）、セキュリティ機微（認証・権限・秘密情報・課金）、影響範囲（共有モジュール・コア抽象・依存元の多さ）、不確実性（plan.md で残決定事項だった箇所・不慣れな領域・実装中に新たな設計判断が要る箇所）。

選択ロジック:

- 高リスクのユニット: 事前宣言で着手前に方向を承認させ、完了後に diff レビューも挟む（前後で挟み込む）
- 中リスクの通常ユニット: 完了後の diff レビュー
- 低リスクの機械的・自己完結・容易に巻き戻せるユニット: 進捗確認だけで素通し

深度がベースラインを調整する。深度は research-plan-annotate が計画時に宣言する minimal / standard / comprehensive の3段階で、plan.md から引き継ぐ。

- minimal: ゲートなしで通し、最後に一度だけ確認
- standard: diff レビューを基準に、高リスクで挟み込み・低リスクで進捗確認に緩和
- comprehensive: 挟み込みを厚めに

plan.md が無い場合のフォールバック経路（3段階の挙動）:

- plan.md がある: それを読みユニット構造をゲート境界に使う（優先経路）
- plan.md が無く作業が小規模: スキル内で軽量なユニット分解を一度挟む。対象コードを読み、ユニット一覧と各ユニットの想定ゲート種別を提示して承認を取り、ゲートループへ入る（research-plan-annotate の minimal 深度に相当する最小計画）
- plan.md が無く作業が実質的（複数ファイル横断・高リスク・ドメイン理解が必要）: その場分解で押し切らず research-plan-annotate を先に通すことを勧める。ユーザーが承知のうえで続行を選べば、その場分解で進めつつ「research による裏付けが無い」ことを明示して始める

research-plan-annotate 側の変更:

Phase 5 Handoff の現行文言「すべてのタスクとフェーズが完了するまで停止しないでください」はユニットゲートと正面から矛盾する。run-to-completion 指示から「新スキルに引き継ぐ」文言に差し替える。これが既存スキル側の唯一の変更。

## 性能・品質・セキュリティ上の前提

各ユニットの検証はプロジェクトのテスト・ビルドコマンドに依存する。コマンドが不明な場合はユーザーに確認する。

ゲートはユーザーの承認待ちで停止する。承認なしに次ユニットへ進まないことを品質保証の要とする。

スキル自体の動作確認は、gh skill 配布の制約上、リポジトリ版をインストール先へ手動同期したうえで新セッションで行う必要がある（スキル本文はセッション開始時にキャッシュされるため）。

## リスクと対策

- 実装セッション中にゲートを素通りするリスク: SKILL.md でゲートを必須ステップとして明示し、承認取得を完了条件に組み込む
- ゲートが過剰でボトルネックになるリスク: 深度とリスクに応じた自動選択で、低リスクユニットは進捗確認に緩和する
- 既存 Handoff との矛盾を放置するリスク: research-plan-annotate の Phase 5 を同時に修正し、両スキルの整合を取る
- 単体実行で research の裏付けを欠いたまま実装するリスク: 実質的な作業では計画スキルへ誘導し、続行時は裏付け欠如を明示する

## 残決定事項（判断ポイント）

### 論点1: 新スキルの名前

リポジトリの命名規約と整合させたい。research-plan-annotate は「パイプラインの動詞チェーン」かつ「human-in-the-loop の機構（annotate）を名前に含む」形。新スキルのパイプラインは実装→ゲート→検証→記録で、human-in-the-loop の機構はゲート。

- A. gated-implementation — 利点: ゲートという特徴を前面に出す。スキルの核心が名前で伝わる / 欠点: 動詞チェーンの規約からは外れる
- B. implement-verify-record — 利点: 動詞チェーンで research-plan-annotate と構造的に対称。3つの成果（実装・検証・結果記録）を表す / 欠点: 核心であるゲートが名前から消える
- C. その他（自由記述）

推奨: A（ゲートがこのスキルの存在理由であり、annotate が計画スキルの核を名前で示すのと同様に、gate を前面に出すのが一貫する）
[Answer]: B（implement-verify-record。動詞チェーンで research-plan-annotate と構造的に対称な命名を優先。ゲートは名前ではなくスキル本文で機構として担保する）
