# AI-DLC概念のresearch-plan-annotate組み込み 実装計画

日時: 2026-06-19
関連調査: `.claude/docs/research/2026-06-19-aidlc-integration-research.md`

## 概要

AI-DLC（AWS Labs）の Inception 側概念のうち、research-plan-annotate と摩擦が小さく価値の高いものを選択的に取り込む。本スキルの責務境界（実装の手前で止め Handoff する）と簡潔さを保ったまま、既存構造を拡張する surgical な変更にとどめる。

なぜこのアプローチか: 調査の結論どおり、両者は設計思想を共有しており新規機構は不要。既存の「残決定事項」「フェーズ省略の非形式的な適応」を、AI-DLC 由来の明示的な型（深度レベル・事前宣言・選択式判断）に格上げするのが最小コストで効果が大きい。

## アプローチ

採用スコープ（残決定事項で確定）:

- 採用 #1 深度レベル（minimal/standard/comprehensive）の明示化
- 採用 #2 着手前のワークフロー宣言（standard/comprehensive のみ宣言、minimal は一言。論点3=B）
- 採用 #3 残決定事項の構造化選択式フォーマット（A/B/C/D + `[Answer]:`、テンプレ直書き。論点2=A）
- 採用 #4 Phase 4 Todo のユニット・グルーピング（論点1=B）
- 見送り #5 NFR レンズ（既存テンプレートでほぼカバー済みのため。論点1=B で対象外）

うち #1 と #2 は密結合（深度は宣言時に選ぶ）なので1つの新セクションとして実装する。#3 はテンプレート2本と Phase 3 の説明追記で実装する。#4 は task-list-template と SKILL.md Phase 4 の一文で実装する。

採用しなかったもの（research より）: Construction/Operations フェーズ、audit/state/session-continuity 機構、常時適用の起動モデル、`aidlc-docs/` 配置、#6 Extensions。理由は research.md 参照。

## 変更内容

### 変更1: SKILL.md に「着手前: ワークフロー宣言と深度選択」を追加（#1 + #2）

対象: `skills/research-plan-annotate/SKILL.md`

挿入位置: `## チェックリスト` のフェーズ番号説明行の直後、`## Phase 0: Design Spec` の前。

```markdown
## 着手前: ワークフロー宣言と深度選択

Phase 0 に入る前に、今回どのフェーズをどの深度で実行するかを宣言し、合意を取る。フェーズ省略をその場の暗黙判断にせず、明示してから始める。

深度は3段階:

- minimal: 設計判断が不要な小修正。Phase 0 を省略し、Research/Plan も要点のみ
- standard: 通常の機能追加・変更。Phase 0〜4 を標準粒度で実行
- comprehensive: 新規プロダクト・高リスク・複数モジュール横断。全フェーズを厚く実行

standard / comprehensive では着手前に宣言する:

> 「今回は <depth> 深度で進めます。実行フェーズ: <該当する Phase>。<省略するフェーズと理由>。この計画でよければ着手します。」

minimal では「設計判断が不要なため Phase 0 を省略し Phase 1 から始めます」の一言で足りる。
ユーザーが深度・対象フェーズの変更を求めた場合は反映してから最初の対象フェーズに進む。
```

あわせて `## 原則` の既存行を深度に接続:

```markdown
- spec.mdは設計の基点。research.md / plan.md の冒頭から参照リンクする。深度 minimal（設計判断が不要な小修正）では Phase 0 を省略してよい
```

### NFR レンズ（#5）について

spec-template.md には既に `## 性能・品質・セキュリティ上の前提` と `## 要件ごとの実装方針 > 非機能要件` があり、plan-template.md には `## 考慮事項`（性能・セキュリティ・後方互換）がある。NFR は実質カバー済みで、追加するなら「観点を明示する一文」程度。費用対効果が低いため既定では見送り、必要時のみ論点1で採用する。

## 影響範囲

- スキルファイルはセッション開始時にキャッシュされる。改修後の動作確認は新セッションで行う（CLAUDE.md）
- 配布版（`~/.claude/skills/...`）は gh skill install のコピーで遅延する。リリースタグはリモート main 同期後に作成する（CLAUDE.md の gh skill 注意点）
- テンプレート変更は後方互換。既存の生成済み spec.md / plan.md には影響しない
- `tests/test_install.sh` は install.sh の回帰テストで、本変更（skills 配下のみ）は対象外。新規テストは不要
- ドキュメント量が増えるため、CLAUDE.md の「いたずらに行数を増やさない」に配慮し挿入は最小限にとどめる

## 考慮事項

- 用語衝突: AI-DLC の「Phase」と本スキルの「Phase 0〜5」が二重定義にならないよう、本スキルの番号体系を正とし、深度語（minimal/standard/comprehensive）のみ借用する
- スコープ膨張: 取り込みを Inception 側に限定し、「don't implement yet」境界を保つ
- 過剰手続き化: minimal 深度では宣言を一言に抑え、小修正の体験を重くしない

## 残決定事項（判断ポイント） — 確定済み

annotation サイクルで確定済み（論点1=B、論点2=A、論点3=B。いずれも推奨どおり）。記録として選択肢を残す。

### 論点1: 採用スコープ（確定: B）

どこまで組み込むか。

- A. コアのみ（#1 深度 + #2 宣言 + #3 選択式）— 利点: 最小・低リスクで効果大 / 欠点: 並列実装の明示やNFR強化は入らない
- B. コア + #4 ユニット・グルーピング — 利点: subagent 並列委譲と接続、Todo の実装効率向上 / 欠点: task-list の記法がやや複雑化
- C. コア + #4 + #5 NFR レンズ — 利点: 非機能観点を明示 / 欠点: #5 は既存とほぼ重複で費用対効果低
- D. その他（自由記述。例: 特定の概念だけ／extensions も検討）

推奨: B（コアの価値を取りつつ、本スキル既存の subagent 並列の発想と自然に接続するため）
[Answer]: B

### 論点2: #3 選択式フォーマットの置き場所（確定: A）

- A. テンプレート内に直書き（references を増やさない）— 利点: シンプル、ファイル数を増やさない / 欠点: 文面の一元管理はできない
- B. `references/decision-format.md` を新設しテンプレ/SKILL から参照 — 利点: 文面を一元管理（similarity-scan-prompts.md と同じ流儀）/ 欠点: ファイルとロード手順が増える

推奨: A（CLAUDE.md のシンプルさ優先。フォーマットは短く、参照ファイル化の便益が小さい）
[Answer]: A

### 論点3: 深度宣言の強制度（確定: B）

- A. 全深度で宣言を必須化（minimal でも明示）— 利点: 一貫性 / 欠点: 小修正が冗長
- B. standard/comprehensive で宣言、minimal は省略の一言のみ — 利点: 小修正の体験を軽く保つ / 欠点: 分岐がやや増える

推奨: B
[Answer]: B
