# handover テンプレート改善 実装計画

日時: 2026-03-27
関連調査: `.claude/docs/research/2026-03-27-handover-granularity-research.md`

## 概要

handover スキルが生成するセッション引き継ぎノートに「リポジトリ状態」セクションを追加し、テンプレートの各所に細かい記述ガイダンスを加える。シンプルさを維持しつつ、次セッションのClaudeがgit状態と作業再開点を即座に把握できるようにする。

## アプローチ

テンプレートへの変更は最小限に抑える:

- セクション追加は1つのみ（リポジトリ状態）
- 既存セクションへの記述指示の補強（次にやること、今回やったこと）
- セッションの目的/背景は「今回やったこと」の冒頭1行として吸収する（独立セクション化しない）

「中途半端な作業状態」と「テスト状態」は「リポジトリ状態」セクションで吸収する。
独立セクションを増やすより、1つのセクションに集約する方が読み手の認知コストが低い。

## 変更内容

### 変更1: 「リポジトリ状態」セクションをテンプレートに追加

対象: `~/.claude/skills/handover/SKILL.md`（テンプレート部分）

「今回やったこと」の直前に挿入する。git状態は「現在何が起きているか」の前提情報であり、実施内容の前に見えるべきだから。

```markdown
## リポジトリ状態

- ブランチ: `feat/xxx`（main から分岐、未マージ）
- 未コミット: `src/foo.ts`（新規）、`README.md`（変更）
- 未プッシュ: 2 コミット（abc1234..def5678）
- テスト: PASS（`bash tests/test_install.sh` 18/18）/ 未実行
```

なし（変更なし・クリーンな状態）の場合は「なし（main、クリーン）」と1行で書く。

### 変更2: 「今回やったこと」にセッション背景の記述指示を追加

現状の指示:
> (動詞+対象+状態で書く。例: ...)

改善後の指示:
> 1行目にセッションの目的・背景を書く（例: 「Issue #5『install.sh CLI引数対応』への対応」「/handover スキルの粒度改善を検討」）。以降は動詞+対象+状態で書く。

冒頭1行でセッション背景をカバーし、独立セクション化を避ける。

### 変更3: 「次にやること」に作業再開起点の指示を追加

現状は「次セッションで必ずやること」の列挙のみ。

改善後: [必須] の冒頭に「作業再開の起点（最初に実行すべきコマンドや確認事項）」を書くよう指示を追加する。

```markdown
## 次にやること

- [起点] git status と bash tests/test_install.sh で現状確認
- [必須] PR #6 をマージする（...）
- [任意] ...
```

`[起点]` タグを新設し、次セッションが最初に何をすべきかを明示する。

### 変更4: Process ステップ4の振り返り項目に Git状態の確認を追加

現状:
> 4. 会話全体を振り返り、以下を意識的に確認する:
>    - 明示・暗示を問わず却下されたアプローチ
>    - セッション序盤に行われた設計判断
>    - エラーや詰まりがあったポイントとその解消方法

追加:
> - 現在のgit状態（ブランチ、未コミット・未プッシュ、テスト状態）

Claudeが「リポジトリ状態」セクションを埋めるための確認手順として明記する。

## 影響範囲

- 既存の引き継ぎノートには影響しない（過去ファイルの変更なし）
- 新テンプレートに従うと「今回やったこと」の1行目が変わる（背景追記）
- 「リポジトリ状態」セクションは `git status` / `git log` 実行が必要になる

## 考慮事項

- 「リポジトリ状態」はhandover生成時点のスナップショット。次セッション開始時には古い可能性があるため、「作業前に git status で確認せよ」という注記を [起点] タグとして表現する
- セクション追加は1つのみに絞り、既存の簡潔さを維持する
- 記入例（SKILL.md末尾のサンプル）も更新して新セクションを示す
