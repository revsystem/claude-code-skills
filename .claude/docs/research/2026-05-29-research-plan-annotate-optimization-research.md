# research-plan-annotate スキル assets分離・最適化 調査レポート

日時: 2026-05-29

## 対象範囲
- `skills/research-plan-annotate/SKILL.md`（全382行）
- 比較対象: `skills/handover/SKILL.md`（96行、先行で重複解消済み）
- 配布経路: `install.sh`（symlink）、`gh skill install`（コピー、CLAUDE.md記載）

## アーキテクチャ概要
本スキルは Research → Plan → Annotate → Todo → Handoff の5フェーズからなる多段ワークフロー。`disable-model-invocation: true` のためモデルの自律起動はせず、`/research-plan-annotate` のスラッシュ明示起動のみで発火する。起動時に SKILL.md 全文がコマンド本文としてコンテキストへ注入され、ワークフローは複数ターンにわたって進行する。

## handover との前提差（最適化方針が逆になる根拠）

| 観点 | handover | research-plan-annotate |
| --- | --- | --- |
| 行数 | 96行 | 382行 |
| 起動 | Stopフックで低コンテキスト時に自動発火 | スラッシュ明示起動のみ |
| 実行スパン | 単一ターンで完結 | 複数ターン（フェーズごと） |
| Read往復の許容度 | 低（緊急時に往復したくない） | 高（熟慮型・各フェーズ境界で自然） |

handover で assets 分離を退けた根拠（低コンテキスト緊急時の Read 往復回避）は本スキルには当てはまらない。本スキルは熟慮型の多段プロセスで、フェーズ境界での Read は安価。assets 分離の評価は handover とは逆向きに傾く。

## 埋め込みテンプレートの棚卸し
SKILL.md 内のコードフェンスは5箇所。手続き本文とは別の「参照素材」であり、各々が特定フェーズでのみ使われる。

| テンプレート | 行 | 概算 | 使用フェーズ |
| --- | --- | --- | --- |
| research.md構成 | 139-173 | 35行 | Phase 1 のみ |
| plan.md構成 | 213-254 | 42行 | Phase 2 のみ |
| 注釈サイクルのASCII図 | 270-276 | 7行 | Phase 3（小さく、本文と一体） |
| タスクリスト構成 | 312-324 | 13行 | Phase 4 のみ |
| result.md付録 | 357-382 | 26行 | Phase 5 の Handoff 文面でのみ言及 |

テンプレート合計は約115行で SKILL.md の約30%。各フェーズでしか使わない素材が起動時に全量常駐し、ワークフロー全ターンを通じてコンテキストを占有する。

## 重要な発見
- 重複なし: handover にあったテンプレート＋記入例の二重骨格は本スキルには存在しない。よって handover と同種の「統合による重複解消」は適用対象外。最適化は専ら「分離による on-demand 化」と「冗長プロセス文の圧縮」の2方向。
- result.md付録が最有力の切り出し候補: 本スキルは「reports は駆動対象外」と明記（21行目・349行目・原則）。result.md付録（26行）はスキルが直接駆動しない素材であり、Handoff 文面からの参照に留まる。スキル本体に常駐させる必然性が最も薄い。
- ASCII図（270-276）は7行と小さく本文と密結合。分離の利得は小さい。
- 残りの手続き本文（類似実装スキャン、subagent委譲判断、注釈反映原則など）はワークフローのロジックそのもので、各フェーズ実行に常時必要。分離対象にならない。

## 既存パターンと規約
- `~/.claude/skills/research-plan-annotate` は本リポジトリへの symlink（ディレクトリごと）。ローカルでは assets サブディレクトリも参照可能。
- リポジトリ内に assets ディレクトリの前例なし（handover も非分離）。assets 参照を行うスキルがゼロのため、分離は新パターンの導入になる。

## 配布仕様の確認結果（当初リスクは解消）
`gh skill install`（CLI ヘルプ）と Agent Skills 仕様（agentskills.io/specification）を確認した。

- スキルはディレクトリであり、`assets/`（テンプレート・リソース）/`references/`（ドキュメント）/`scripts/`（実行コード）が正式なオプションサブディレクトリとして定義されている。
- プログレッシブディスクロージャーが仕様の中核: SKILL.md 本文は起動時に全量ロードされるが、`assets/`・`references/` 配下は「必要になったときだけ」ロードされる。仕様は「SKILL.md は500行未満に保ち、詳細な参照素材は別ファイルへ移す」と明示的に推奨。
- ファイル参照はスキルルートからの相対パス、1階層まで（`references/REFERENCE.md` 等）。
- `gh skill install` は本仕様に準拠しスキルディレクトリを丸ごとコピーする（ローカルは files are copied、リポジトリ install も同梱リソースを配布）。

結論: assets 分離は仕様公認の手法で配布も担保される。当初「最大の不確実性」とした配布リスクは消える。配布方式（gh skill install / symlink）を変更する必要もない。テンプレート群は仕様上 `assets/`（document templates）が意味的に正しい配置先。

## 注意点・リスク
- on-demand 化の弱点: Claude がフェーズ到達時に assets を Read せず記憶からテンプレートを再構成すると、出力テンプレートがドリフトする。各フェーズの該当箇所に明示的な Read 指示（相対パス）を埋める必要がある。
- 分離の前例がリポジトリ内にゼロ。新パターン導入として install.sh のテスト（test_install.sh）や運用ドキュメント（CLAUDE.md）への波及有無を確認する。
- スキルファイルはセッション開始時にキャッシュされる（CLAUDE.md記載）。編集後の挙動確認は新セッションで行う。

## 結論（次フェーズへの示唆）
本スキルの最適化は handover と異なり「重複解消」ではなく「フェーズ局所テンプレートの on-demand 化」と「冗長プロセス文の圧縮」が軸。配布リスクが解消したため、assets 分離は仕様的にも費用対効果的にも有力（382行→本文圧縮＋テンプレート外出し）。最有力は (1) 5テンプレートを `assets/` へフル分離、(2) out-of-scope の result.md付録のみ切り出し、の2系統。具体案（フル分離／付録のみ分離／インライン圧縮併用／現状維持）は plan.md の残決定事項で案出しする。
