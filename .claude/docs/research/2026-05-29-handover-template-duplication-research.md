- 比較対象: `skills/research-plan-annotate/SKILL.md`（テンプレート表現の参照実装）

## 現状構造

handover の SKILL.md は同じ8セクションの骨格を2回提示している。

| ブロック | 行 | 内容 |
| --- | --- | --- |
| テンプレート | 25-62 | 8セクションの雛形。各セクションに括弧内のヒント（書き方＋`例:`）を併記 |
| Quality Rules | 64-71 | 一部セクションの before→after 例（体言止め禁止など） |
| 記入例 | 75-136 | 同じ8セクションを handover スキル自身の開発実話で充填した完成例 |

8セクション = リポジトリ状態 / 今回やったこと / 決定事項 / 捨てた選択肢と理由 / ハマりどころ / 学び / 次にやること / 関連ファイル。

## 重複の所在
- セクション見出しの列挙がテンプレートと記入例で完全に重複（8見出し×2）。
- 「書き方の指示」が3か所に分散: テンプレートの括弧内ヒント、Quality Rules、記入例の実例。読み手（Claude）は同じ意図を3形態で受け取る。
- 記入例ブロックだけで約62行。SKILL.md の約45%を占める。

## 重複が生む問題
- 編集コスト: セクション追加・改名のたびに2か所（＋Quality Rules）を同期させる必要がある。同期漏れで雛形と例が食い違うリスク。
- コンテキスト消費: handover は context 逼迫時（compaction直前・70%閾値）に inline 実行される。invoke 時に全文が展開されるため、重複分がそのまま無駄になりやすい。

## 参照実装（同リポジトリ）
- research-plan-annotate は「構成」内に `（コードスニペット）` 等のプレースホルダと `例:` をインライン併記し、満杯の完成例を別掲しない方式。
- この方式は骨格の二重定義を避けつつ、各セクションの直下にヒントを置く。handover の改善方針の有力な土台になる。

## 既存パターンと規約
- `assets/` ディレクトリはリポジトリ内に未使用。`gh skill install` はファイルをコピーする（CLAUDE.md記載）ため、分離ファイルを増やすと配布物が増える。
- 直前の別セッションでの結論: 137行は小さく、`assets/` 分離による progressive disclosure の利得は限定的。低コンテキスト時に Read 往復が増える・Claude が例の読み込みを省略する懸念がある。→ 分離より「統合・圧縮」が有力。

## 注意点・リスク
- 記入例は few-shot として出力のトーン・粒度を支えている。単純削除は品質低下リスク。完成例の「実話の具体性」は残す価値がある。
- Quality Rules の before→after 例は記入例とは別機能（アンチパターン提示）。統合対象に含めるか要判断。

## 結論（次フェーズへの示唆）
重複の本質は「8セクションの骨格を雛形と記入例で二重に持つこと」。改善方向は分離（assets化）ではなく統合が適切。具体案は plan.md の残決定事項で案出しする。
