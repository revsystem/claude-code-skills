# handover スキルへの外部引き継ぎ設計の知見取り込み 調査レポート

日時: 2026-05-29

## 対象範囲

- `skills/handover/SKILL.md`（96行、統合済み）
- 参照（比較対象）: 外部の多セッション調査ノート用スキル（参考実装）。

## 設計モデルの差（取り込み範囲の前提）

- handover: 1セッション = 1スナップショットファイル（`.claude/handovers/YYYY-MM-DD_HHmm.md`）。低コンテキストで自動起動。履歴 = 日付ファイルの列。
- 参考実装の引き継ぎファイル: 1トラック（調査フォルダ）= 1つの生きた引き継ぎファイル。セッション跨ぎで更新し、内部の追記ログに記録。

結論: 参考実装のトラック機構・命名規約・調査種別は別問題を解くもので handover には非適用。取り込むのは「引き継ぎファイル単体の中身の設計」に限る。

## 取り込む3点（採用）

1. Cold-read order（最有力）: 「予備知識ゼロで開いた人がこの順で読めば最速で文脈復元できる」読書動線を所要時間目安付き（≈3分/≈10分）で並べる。handover には読書順序の明示がない（関連ファイルはパス＋一言、次にやること[起点]は最初の操作のみ）。読み手が次セッションのClaudeである handover に直接効く。
2. Status / Drives 行: 冒頭に Status（Working/In review/Blocked/Done）と駆動 Issue/PR リンクを1〜2行。handover は日時のみ。CLAUDE.md の「PRはIssueに紐付け」運用と整合し、再開時の立ち位置が即わかる。
3. Memory references 行: `~/.claude/.../memory/` の関連エントリへのポインタ。本リポジトリにもメモリ機構があり、次セッションの関連知識への最短到達を助ける。

## 見送り（非採用）とその理由

- トラック機構 / ファイル命名バンド / 調査種別: 別問題を解く機構。handover の per-session モデルに不要。

## 既存パターンと規約

- handover の現行8セクション: リポジトリ状態 / 今回やったこと / 決定事項 / 捨てた選択肢と理由 / ハマりどころ / 学び / 次にやること / 関連ファイル。各見出し直下に `<!-- -->` の書き方ルール（前回の統合で確立）。
- 取り込みは新セクション乱立を避け、既存セクションへの統合を優先する（handover を小さく保つ方針＝assets 非分離の判断と整合）。

## 注意点・リスク

- handover は低コンテキスト自動起動。セクション追加で記入負荷が増えると、肝心のタイミングで生成が重くなる。3点は「軽い1〜数行」に収め、必須化より任意寄りにするか要検討（残決定事項）。
- 参考実装は外部由来。汎用パターンのみ反映し、外部固有の組織名・具体例は持ち込まない。
- スキルはセッション開始時にキャッシュ。編集後の挙動確認は新セッション。

## 結論（次フェーズへの示唆）

3点はいずれも「引き継ぎファイル単体の中身」を強化する汎用パターンで handover に適合。設計の論点は (a) 各点を新セクションにするか既存セクションへ統合するか、(b) 必須か任意か、(c) Cold-read order と既存「関連ファイル」「次にやること[起点]」の重複整理。plan.md の残決定事項で確定する。
