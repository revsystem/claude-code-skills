## 捨てた選択肢と理由

- handover の assets 分離: 低コンテキスト自動起動時に Read 往復が増える・例の読み飛ばしリスク — 単一ブロック統合を採用
- research-plan-annotate の手続きロジック（スキャン手順・subagent委譲）の references 外出し: 毎回必須の素材は on-demand 化しても Read が増えるだけ — インライン維持
- research-plan-annotate の references 不要案（推奨A）: ユーザー判断で案B採用。確認フロー文面は条件起動素材のため外出しの意味があり、ファイル増コストは許容
- disable-model-invocation:false 化: 重い5フェーズが軽微作業でも過剰起動するリスク・description 厳格化が別途必要 — true 維持

## 学び

- Agent Skills 仕様: assets/（テンプレート）・references/（オンデマンド文書）・scripts/ が正式サブディレクトリ。本文は activate 時に全量ロード、サブディレクトリは必要時のみ。「SKILL.md は500行未満、詳細は別ファイルへ」が公式推奨。ファイル参照はスキルルートから1階層
- gh skill install はこの仕様準拠でスキルディレクトリを丸ごとコピーし、同梱リソースも配布する。ローカルの ~/.claude/skills/* は本リポジトリへの symlink（assets も参照可能）
- skills frontmatter の disable-model-invocation:true はスラッシュ明示起動のみに限定し自律起動を禁止する。description は自律起動の判定に使われるため、false 化時は「いつ使わないか」を description に畳み込まないと過剰起動する
- 最適化方針はスキルのサイズと起動特性で逆になる: 低コンテキスト自動起動・小サイズ（handover）は統合してインライン、熟慮型多段・大サイズ（research-plan-annotate）は分離してオンデマンド
