## 決定事項

- terraform-code-reviewer は read-only（`Read, Grep, Glob`）を採用。ブログの「コードレビューは read-only サブエージェント」原則を優先
- read-only レビューポリシーは README に推奨スニペットとして同梱する方式を採用。グローバル `~/.claude/CLAUDE.md` への直接追記やプロジェクト CLAUDE.md への追記は行わない
- コミットは原子的に2分割（エージェント+README / スキル）
- リモートブランチは削除せず残す

## 捨てた選択肢と理由

- terraform-code-reviewer に Write/Edit を残す案（レビューと修正適用の二役）: ブログの read-only レビュー原則と衝突し、レビュー役がファイルを書き換えると客観性が薄れる — read-only（Read, Grep, Glob）を採用
- ポリシーをグローバル `~/.claude/CLAUDE.md` に直接追記する案: リポジトリ外の個人設定を編集することになる — README の推奨スニペット同梱を採用（ユーザー選択）
- ポリシーをプロジェクト CLAUDE.md に追記する案: この repo はオーサリング用で .tf を扱わず効果が薄い — 不採用（ユーザー選択）

## ハマりどころ

## 学び

- ブログの read-only レビュー構成の具体例は `Read, Grep, Glob`。レビューエージェントに Grep/Glob がないとプロジェクト横断のファイル探索ができず、Read で明示パスを開くしかなくなる
- terraform-code-reviewer の Write/Edit は設計ドキュメント（`.claude/docs/superpowers/specs/2026-03-18-terraform-code-reviewer-agent-design.md:98`）で「提案の適用まで担う」意図で入れられていた。今回 read-only 化でこの設計判断を反転させた（spec は履歴記録として未更新、判断根拠はコミットメッセージに記載）
