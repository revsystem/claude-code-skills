## 決定事項

- terraform-code-reviewer（skills/user agent）はベア名 allowlist `mcp__<server>` で MCP を許可する。read-only allowlist を維持
- plugins リポジトリ側の同等エージェントは `disallowedTools: Write, Edit` を採用（ユーザー判断・plugins 側で適用済み）。MCP を自動継承しつつ read-only を保つため
- 使い分けの根拠は CLAUDE.md に記録する（将来同じ判断を再導出しないため）

## 捨てた選択肢と理由

- skills 側で `disallowedTools: Write, Edit` を使う案: MCP 以外の継承ツール（WebFetch/WebSearch 等）も広く入り、絞った allowlist 設計から外れる — user agent ではベア名 allowlist を採用
- plugin 側で `mcp__plugin_<plugin>_<server>` をベア名 allowlist にする案: プラグイン名（marketplace 名）に結合し rename や別マーケット名インストールに弱い — plugin では `disallowedTools` を採用

## ハマりどころ

- `tools: Read, Grep, Glob` を指定した時点でインライン定義した MCP サーバーのツールが呼べなくなる → `tools` allowlist は MCP を除外する仕様（mcpServers は接続のみ担当） → allowlist に `mcp__<server>` を明示して解消

## 学び

- サブエージェントの `tools` allowlist を指定すると MCP ツールは列挙しない限り除外される（公式 sub-agents ドキュメント「The subagent can't ... use any MCP tools」）。`mcpServers` はサーバー接続、`tools` allowlist は呼び出し可否ゲートで役割が別
- MCP 許可ルールの表記（permissions ドキュメント）: `mcp__<server>` でそのサーバー全ツール、`mcp__<server>__*` も全ツール、`mcp__<server>__<tool>` で個別。この形式は subagent の `tools`/`disallowedTools` でも共通
- plugin 提供の MCP ツール名は `mcp__plugin_<プラグイン名>_<サーバー名>__<ツール>` と名前空間化される（実ツール名で確認）。allowlist にすると plugin/marketplace 名に結合して脆い
- `disallowedTools` は継承プールから除外する denylist。`tools` と併用すると disallowedTools が先に適用され、残りに対して tools が解決される
- 公式ドキュメントは `https://docs.claude.com/en/docs/claude-code/sub-agents` → `https://code.claude.com/docs/en/sub-agents` に 301 リダイレクトする（WebFetch はリダイレクト先を再取得する必要あり）
