## 決定事項

- CLI 引数は `TYPE:NAME` 形式のみサポートする。TYPE だけの指定（`skills` 等の一括指定）はサポートしない。ユーザーの「実装が複雑すぎる」というフィードバックにより簡素化
- `set -eo pipefail` を使用し `set -u` は除外する。`set -u` は bash 配列処理を複雑にする（`"${arr[@]+"${arr[@]}"}"` パターンが必要になる）ため
- hooks 依存関係は SKILL.md frontmatter の `hooks:` フィールドで宣言する。外部設定ファイルや命名規則ではなく、skill 定義ファイル内に co-locate する方式
- 依存方向は「hook が skill に依存」（skill をインストールすると依存 hooks も自動インストール）
- docs は `.claude/docs/` に集約する。`docs/` ディレクトリは廃止

## 捨てた選択肢と理由

- TYPE だけの一括指定（`skills`, `agents`, `hooks`）: 実装が複雑になる（ALL フラグ、優先順位ルール、dedup ロジック）— `TYPE:NAME` 形式のみに簡素化
- `--skills NAME` フラグ形式（アプローチ A）: getopt 相当の処理が必要でシェルスクリプトとして複雑 — コロン区切り positional args（アプローチ B）を採用
- カテゴリフラグ + 名前フラグ分離（アプローチ C）: フラグ数が増えて冗長 — アプローチ B を採用
- 命名規則で hooks 依存を推定する案: 暗黙的で柔軟性がない — SKILL.md frontmatter の `hooks:` フィールドで明示宣言する方式を採用
- リポジトリルートの一括定義ファイル（deps.json 等）: 依存情報が skill 定義から離れる — SKILL.md 内に co-locate する方式を採用
- `set -euo pipefail`（-u 付き）: 空配列の扱いが `"${arr[@]+"${arr[@]}"}"` のように複雑になる — `-u` を除外して `set -eo pipefail` を採用
- include + exclude 両方サポート: 複雑になる — include のみに簡素化

## ハマりどころ

- `((PASS++))` が `set -e` 下で PASS=0 のときクラッシュする → `((expr))` は結果が 0 のとき exit code 1 を返す仕様 → `PASS=$((PASS + 1))` に変更
- awk の `!/pattern/` が zsh で動作しない → zsh の `!` ヒストリ展開と衝突する → else パターン `f==1 && h{h=0}` に書き換えて解決
- `rm` がシンボリックリンク削除時に確認プロンプトを出す → `rm -i` エイリアスが設定されている → `rm -f` を使用して解決
- スペックレビューで ALL フラグと名前リストの混在時の優先順位が未定義と指摘された → 仕様自体を簡素化して ALL フラグを廃止した

## 学び

- bash の `((expr))` は式の評価結果が 0 のとき exit code 1 を返す。`set -e` との組み合わせでサイレントクラッシュを起こす。`var=$((var + 1))` が安全
- awk スクリプト中の `!` は zsh のヒストリ展開と衝突する。否定パターンを避けて else 分岐で書くのが安全: `!/^  /{h=0}` → `f==1 && h{h=0}`
- superpowers の brainstorming → spec → spec review → plan → plan review のフルサイクルはスペックの品質を上げるが、ユーザーが「複雑すぎる」と感じたらサイクル途中でも仕様を見直すべき
- `CLAUDE_CODE_DIR` 環境変数で install.sh の出力先をオーバーライドする設計はテスタビリティに有効
