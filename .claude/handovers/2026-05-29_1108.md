## 決定事項

- 重複解消は assets 分離ではなく単一ブロックへの統合方式を採用。理由: 137行と小さく progressive-disclosure 利得が薄い／handover は低コンテキスト時に inline 実行されるため Read 往復を増やしたくない
- ヒントの埋め込みは HTML コメント方式（案A）。括弧書き併記より meta と content の分離が明確で、消し忘れても GitHub プレビューでは非表示で読み手に無害
- コメント混入の緩和として、統合ブロック冒頭に「複製して実ノートを書く際はコメント行を削除する」と明記
- 統合見出し名は「記入例（このまま雛形として使う）」（複製可能であることが一読で伝わる）

## 捨てた選択肢と理由

- assets/EXAMPLE.md への分離: 配布ファイル増・低コンテキスト時の Read 往復増・Claude が例を読み飛ばすリスク — 単一ブロック統合を採用
- 記入例の単純削除: few-shot のトーン・粒度ガイドを失い出力品質が落ちる — 満杯の実例（handover 開発実話）を温存
- 括弧書き注記の併置（案B、現テンプレート踏襲）: 実例本文と指示が同じ箇条書きレベルで混ざり、消し忘れ時にノート本文に露出する — HTML コメント（案A）を採用

## 学び

- handover はテンプレートを逐語コピーするのではなく「記入例の構造を真似た新規ノートを生成」するのが期待動作。HTML コメントは Claude への指示として読まれ、生成ノートには再現されない前提（ただし機械的コピー時は残りうるため緩和文を入れた）
- HTML コメントは SKILL.md のコードフェンス（```markdown）内では表示されるが、生成された実ノートを GitHub/VS Code でプレビューすると非表示になる二層構造。利点・欠点はこの差から生じる
- このリポジトリのもう一方のスキル research-plan-annotate は「構成にプレースホルダと例をインライン併記し満杯の完成例を別掲しない」方式で、今回の統合方針の参照実装になった
