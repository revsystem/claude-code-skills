# Research → Plan → Annotate ワークフロー

コードを書く前に「調べる→計画する→注釈で磨く」を徹底するスキル。
plan.mdのタスクリスト完成後、ユーザーの指示で実装フェーズに移行する。

## いつ使うか

- 既存コードベースへの機能追加・変更
- バグ調査で原因箇所が不明なとき
- 複数ファイルにまたがる変更
- ドメイン知識やシステム理解が必要な作業

単一ファイルの軽微な修正には不要。

reports（成果記録）は本スキルの駆動対象外。実装・検証完了後に記録する工程であり、多くの場合は別セッションで行われる。本スキルは Handoff で reports の作成を次工程に申し送る。

## チェックリスト

以下のタスクを順番に完了する:

1. Design Spec: 設計判断を spec.md にまとめ `.claude/docs/specs/` に保存する
2. Research: コードベースを深く読み、research.mdを書く
3. Plan: research.mdを踏まえてplan.mdを書く
4. Annotate: ユーザーの注釈を反映してplan.mdを更新する（1〜6回繰り返し）
5. Todo: plan.mdにタスクリストを追加する
6. Handoff: 実装フェーズへの引き継ぎ

各ステップはフェーズ番号で対応する（Design Spec=Phase 0、Research=Phase 1、Plan=Phase 2、Annotation=Phase 3、Todo=Phase 4、Handoff=Phase 5）。

## 着手前: ワークフロー宣言と深度選択

Phase 0 に入る前に、今回どのフェーズをどの深度で実行するかを宣言し、合意を取る。フェーズ省略をその場の暗黙判断にせず、明示してから始める。

深度は3段階:

- minimal: 設計判断が不要な小修正。Phase 0 を省略し、Research/Plan も要点のみ
- standard: 通常の機能追加・変更。Phase 0〜4 を標準粒度で実行
- comprehensive: 新規プロダクト・高リスク・複数モジュール横断。全フェーズを厚く実行

standard / comprehensive では着手前に宣言する:

> 「今回は <depth> 深度で進めます。実行フェーズ: <該当する Phase>。<省略するフェーズと理由>。この計画でよければ着手します。」

minimal では「設計判断が不要なため Phase 0 を省略し Phase 1 から始めます」の一言で足りる。
ユーザーが深度・対象フェーズの変更を求めた場合は反映してから最初の対象フェーズに進む。

## Phase 0: Design Spec

目的: 実装前の設計判断（何を・なぜ作るか）を spec.md として固定し、Research/Plan の前提を明確にする。コードの読解より上位の「プロダクト/機能の設計」を扱う。

### いつ spec.md を書くか

- 新規プロダクト・新機能・大きめの仕様変更で、設計判断（スコープ・アーキテクチャ・技術選定）が必要なとき
- 既存コードへの小さな修正で設計判断が不要なときは Phase 0 を省略し、Phase 1 から始めてよい

### ユーザー提供の設計書がある場合

ユーザーが既に設計書（開発計画書など）を書いている場合は、新規生成せず既存内容を採用する。所定の保存先（後述）に無ければそこへ移動・整形し、不足セクションのみ補う。勝手に設計を変えない。

### spec.md の進め方

設計判断を文書化する。推測ではなく、ユーザーの要求と制約に基づいて書く。判断に迷う点は独断で1案に決めず「残決定事項」に出してから Annotation で確定させる（Plan と同じ原則）。

含める観点:

- 目的・コンセプト、想定利用者・利用シーンと制約（最重要前提があれば明記）
- スコープ（含める／含めない）とその根拠
- アーキテクチャ設計（全体構成・レイヤ・主要な構造判断の理由）
- 技術選定と根拠（代替案と不採用理由）
- データ設計、要件ごとの実装方針
- 性能・品質・セキュリティ上の前提、リスクと対策

### spec.md の保存先

プロジェクトルートの `.claude/docs/specs/` に `YYYY-MM-DD-<topic>-spec.md` として保存する。
構成は `assets/spec-template.md` を読み、その雛形に従って記述する。「残決定事項」セクションは任意で、判断ポイントが無ければ省略する。

### Design Spec 完了の基準

spec.md をユーザーに提示し、確認を求める:

> 「spec.md を `.claude/docs/specs/` に書きました。内容を確認してください。設計判断に修正・追加があれば指摘してください。問題なければ調査フェーズに進みます。」

ユーザーの承認を得るまで Phase 1 に進まない。

## Phase 1: Research

目的: 関連コードを深く理解し、持続的なアーティファクトとして記録する。
research.md の冒頭に、対応する spec.md（`.claude/docs/specs/...`）へのリンクを記載する。

### 類似実装スキャン（Research開始前の必須ステップ）

Researchの深掘りに入る前に、依頼内容に類似した実装がコードベース内にないか確認する。
重複実装を防ぎ、既存コードの再利用機会を発見するために毎回実施する。

#### スキャン手順

1. 依頼内容からキーワードを抽出する
   - 機能名・ドメイン語（notify, export, auth, cache など）
   - 関連エンティティ名（User, Order, Session など）
   - 動詞（send, parse, validate, convert など）

2. コードベースをgrep/findで横断検索する
   - 関数名・クラス名・モジュール名
   - ファイルパス・ディレクトリ名
   - 関連するコメント

3. 類似度を3段階で評価する
   - 高: 既存実装で要件をほぼ満たせる（同じ機能が実質的に存在する）
   - 中: 既存実装を流用・拡張すれば対応できる
   - 低: 類似実装なし、新規実装が必要

#### ユーザーへの確認フロー

類似実装が見つかった場合、Research深掘りの前に確認を求める。確認なしで深掘りに進まない。
確認に使う高/中/低類似度の文面は `references/similarity-scan-prompts.md` を読み、その文面で方針を確認する。

ユーザーから方針A（既存をそのまま使う）を選択された場合、Phase 2以降は省略してHandoffに進む。

### Subagentへの委譲判断（Research開始前に評価する）

Researchを開始する前に、調査の性質を評価してsubagentに委譲するか判断する。ユーザーに確認せず、この基準で自律的に判断する。

**`explore` subagentに委譲する（探索的な調査）:**
- 調査対象ディレクトリが複数にまたがる、またはファイルパスが事前に不明
- 読んだ後に直接編集するのではなく、「理解すること」が目的
- 複数の独立したテーマを並列調査できる
- 外部ドキュメント・Web検索が必要な調査

**親セッションで直接実施する（目的的な読み込み）:**
- 読んだ直後に編集するファイル（StrReplaceのold_stringを正確に取得するため）
- パスが自明で1〜2ファイルの確認にとどまる
- 直前の会話コンテキストが調査方針に影響する場合
- 類似実装スキャン（grepによる横断検索）

**subagentに委譲する場合の手順:**
1. 調査指示を具体的に言語化する（対象ディレクトリ、調査観点、成果物パス）
2. 独立したテーマが複数あるかを判断する:
   - 単一テーマ → `explore` subagentを1つ起動する
   - 独立した複数テーマ（例: APIエンドポイント／DBスキーマ／認証フロー）→ テーマごとに `explore` subagentを同時に起動し、並列調査する
3. 各 `explore` subagentに以下を指示する:
   - 調査対象: 具体的なディレクトリ・モジュール名
   - 調査観点: 何を理解したいか（アーキテクチャ、データフロー、エラーハンドリング等）
   - 成果物: `.claude/docs/research/YYYY-MM-DD-<topic>-research.md` に書く
   - 返答形式: ファイル全文ではなく、要約と発見事項を返す（親セッションのコンテキストを圧迫しないため）
4. subagent（並列起動した場合は全て）が完了したらresearch.mdを読み、Planフェーズに進む

> **注意**: subagentへの指示が曖昧だと的外れなresearch.mdになる。「何を・どこを・なぜ調べるか」を具体的に書く。並列起動する場合は、各テーマが互いに独立している（共有状態や順序依存がない）ことを確認してから起動する。

### 調査の進め方

対象のフォルダ・モジュールを詳細に読み、仕組み・機能・すべての詳細を理解する。表面的な読解では不十分。理解が終わったら、学習内容と発見事項の詳細なレポートをresearch.mdに書く。

ユーザーが調査対象を指定していない場合は、先に確認する:

> 「どのフォルダ/モジュールを重点的に調査しますか？ 具体的なパスがあれば教えてください。」

調査時のチェック項目:

- 関数シグネチャだけでなく、呼び出し元・呼び出し先を追う
- エラーハンドリングやエッジケースの扱いを確認する
- 既存のキャッシュ層、ミドルウェア、共通パターンを把握する
- 暗黙の前提や規約（ORM規約、命名規約、ディレクトリ構造の慣習）を特定する

### research.mdの構成

プロジェクトルートの `.claude/docs/research/` に `YYYY-MM-DD-<topic>-research.md` として保存する。
構成は `assets/research-template.md` を読み、その雛形に従って記述する。

### Research完了の基準

research.mdをユーザーに提示し、以下を確認する:

> 「research.mdを書きました。内容を確認してください。理解に誤りや不足があれば指摘をお願いします。問題なければ計画フェーズに進みます。」

ユーザーの承認を得るまでPhase 2に進まない。

## Phase 2: Plan

目的: 実装アプローチを具体的に設計し、コードスニペット付きの計画文書を作る。

### 計画の進め方

plan.mdを書く前に、変更対象となるソースファイルを実際に読む。読んでいないファイルについて変更を提案しない。実際のコードベースに基づいて計画を立てる。推測で書かない。

計画作成時点でユーザー判断が必要と分かる選択肢は、Claude が独断で 1 案に決めず、「残決定事項」セクションに案出しして annotation サイクルで確定させる。

計画文書の要件:

- 変更対象のファイルパスを明記する
- 主要な変更内容をコードスニペットで示す
- トレードオフや代替案を記載する
- 影響範囲（既存テスト、API、他モジュール）を明記する

### 参照実装の活用

類似機能がOSSや同一コードベース内にある場合、参照として取り込む。
ゼロから設計するよりも、実績のある実装を土台にしたほうが品質が上がる。

ユーザーが参照実装を提示した場合は、それを計画に織り込む:

> 「提示いただいた実装を参考に、当プロジェクトの規約に合わせたアプローチをplan.mdに記載します。」

### plan.mdの構成

プロジェクトルートの `.claude/docs/plans/` に `YYYY-MM-DD-<feature-name>-plan.md` として保存する。
構成は `assets/plan-template.md` を読み、その雛形に従って記述する。ヘッダの「深度」には着手前に宣言した深度（minimal / standard / comprehensive）を記録し、実装フェーズ（implement-verify-record）がゲート粒度の判断に読み取れるようにする。「残決定事項」セクションは任意で、判断ポイントが存在しない場合は省略する。

### Plan完了の基準

plan.mdをユーザーに提示し、注釈サイクルへの移行を促す:

> 「plan.mdを書きました。内容を確認し、修正・追加したい箇所にインラインで注釈を入れてください。注釈を反映してplan.mdを更新します。」

## Phase 3: Annotation Cycle

目的: ユーザーの判断・ドメイン知識をplan.mdに注入し、計画の精度を上げる。

### 注釈サイクルの流れ

```
ユーザーがplan.mdに注釈を追加
  → Claudeが注釈を反映してplan.mdを更新
  → ユーザーが再確認
  → （1〜6回繰り返す）
  → ユーザーが承認
```

### ユーザーへの案内

注釈の入れ方をユーザーに説明する:

> plan.mdを開いて、修正したい箇所に直接メモを書き込んでください。例:
>
> - 「ここはPATCHではなくPUTを使う」
> - 「このセクション不要、削除」
> - 「drizzle:generateでマイグレーション生成、生SQLは使わない」
> - 「このフィールドはリストではなくアイテムに持たせる。スキーマセクションを修正」
>
> 注釈を入れたら教えてください。内容を反映してplan.mdを更新します。

### 注釈反映時の原則

- 注釈の意図を正確に汲み取る。不明な場合は確認する
- ユーザーが判断に迷っている、または質問形式の注釈を入れた場合は、単一の答えや open question で返さず、複数案（メリット・デメリット・推奨）を提示してから判断を仰ぐ。提示後も実装には進まない（注釈サイクルを継続する）
- 注釈に従って計画を修正し、コードスニペットも更新する
- 反映後に「まだ実装しません」の姿勢を維持する
- 反映完了後、再確認を促す:

> 「注釈を反映しました。再度確認をお願いします。追加の修正があればまた注釈を入れてください。問題なければタスクリストを作成します。」

### 構造化された判断ポイント（残決定事項の選択式）

spec.md / plan.md の「残決定事項」は選択式（A/B/C/D…）で提示し、各案に利点・欠点・推奨を付す。ユーザーは該当論点の `[Answer]:` 行に記号を書くか、自由記述（その他）で答える。自由記述のインライン注釈と併用してよい。`[Answer]:` が埋まった論点は確定として計画へ反映する。

### 注釈サイクルの完了条件

ユーザーが「問題ない」「OK」「進めて」等の承認を出した時点で完了。
承認がない限り、実装に進まない。

## Phase 4: Todo List

ユーザーの承認後、計画を完了するために必要なすべてのフェーズと個々のタスクを含む詳細なToDoリストをplan.mdに追加する。まだ実装しない。

フェーズ（大区分）とタスク（個別ステップ）の2階層で構成する。各タスクは2〜5分で完了できる単位に分割する。
タスクは独立実装可能なユニット単位でまとめ、並列化可能なユニットを明示する（subagent 並列委譲に接続）。
書式は `assets/task-list-template.md` を読み、その雛形に従う。

## Phase 5: Handoff

全フェーズ完了後、以下のメッセージで実装フェーズへ引き継ぐ:

> Design Spec → Research → Plan → Annotateが完了しました。
>
> - 設計仕様: `.claude/docs/specs/YYYY-MM-DD-<topic>-spec.md`
> - 調査レポート: `.claude/docs/research/YYYY-MM-DD-<topic>-research.md`
> - 実装計画: `.claude/docs/plans/YYYY-MM-DD-<feature-name>-plan.md`
>
> plan.md にユニット・グルーピング済みのタスクリストが入っています。実装は implement-verify-record スキルで、ユニットごとにゲート（実装 → 検証 → レビュー → 承認）を挟みながら進めます。次のように開始してください:
>
> 「/implement-verify-record この計画の実装を開始して」
>
> implement-verify-record は承認なしに次のユニットへ進みません。全ユニット完了後、検証結果を `.claude/docs/reports/YYYY-MM-DD-<topic>-result.md` に記録します。

## 原則

- 設計・調査・計画はいずれもファイルに書く。チャット上の口頭説明だけで済ませない
- 用途別ディレクトリに保存する: 設計仕様は `.claude/docs/specs/`、調査は `.claude/docs/research/`、計画は `.claude/docs/plans/`
- spec.mdは設計の基点。research.md / plan.md の冒頭から参照リンクする。深度 minimal（設計判断が不要な小修正）では Phase 0 を省略してよい
- 「don't implement yet」を徹底する。承認前の実装は禁止
- auto/bypass・無人実行とは非互換。承認ゲートと「don't implement yet」はツール承認の自動化とは別レイヤの人手の関門であり、auto の非中断圧力より優先する。承認できる人間がいなければゲートを越えて実装に進まない
- 注釈サイクルがこのスキルの核心。ユーザーの判断を計画に織り込む工程を省略しない
- 参照実装がある場合は積極的に活用する
- research.mdは計画の根拠。plan.mdから参照できるようリンクする
- 類似実装スキャンはResearch深掘りの前の必須ステップ。重複実装を防ぐために毎回実施する
- reports（成果記録）は本スキルの駆動対象外。Handoff で申し送るにとどめ、スキルが直接 reports を作成しない

## 付録: result.mdのテンプレート（参考）

reports は検証結果・ポストマージレビュー・デプロイ結果など種類が多様なため、テンプレートは「必須の型」ではなく「参考の出発点」。プロジェクトの実態に合わせて自由に変形してよい。
雛形が必要なら `assets/result-template.md` を読む。保存先はプロジェクトルートの `.claude/docs/reports/` に `YYYY-MM-DD-<topic>-result.md`。
