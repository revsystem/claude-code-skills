# Research → Plan → Annotate ワークフロー

コードを書く前に「調べる→計画する→注釈で磨く」を徹底するスキル。
plan.mdのタスクリスト完成後、ユーザーの指示で実装フェーズに移行する。

## いつ使うか

- 既存コードベースへの機能追加・変更
- バグ調査で原因箇所が不明なとき
- 複数ファイルにまたがる変更
- ドメイン知識やシステム理解が必要な作業

単一ファイルの軽微な修正には不要。

## チェックリスト

以下のタスクを順番に完了する:

1. Research: コードベースを深く読み、research.mdを書く
2. Plan: research.mdを踏まえてplan.mdを書く
3. Annotate: ユーザーの注釈を反映してplan.mdを更新する（1〜6回繰り返し）
4. Todo: plan.mdにタスクリストを追加する
5. Handoff: 実装フェーズへの引き継ぎ

## Phase 1: Research

目的: 関連コードを深く理解し、持続的なアーティファクトとして記録する。

### 類似実装スキャン（Research開始前の必須ステップ）

Researchの深掘りに入る前に、依頼内容に類似した実装がコードベース内にないか確認する。
重複実装を防ぎ、既存コードの再利用機会を発見するために毎回実施する。

#### スキャン手順

1. 依頼内容からキーワードを抽出する
   - 機能名・ドメイン語（notify, export, auth, cache など）
   - 関連エンティティ名（User, Order, Session など）
   - 動詞（send, parse, validate, convert など）

2. コードベースをgrep/findで横断検索する
   - 関数名・クラス名・モジュール名
   - ファイルパス・ディレクトリ名
   - 関連するコメント

3. 類似度を3段階で評価する
   - 高: 既存実装で要件をほぼ満たせる（同じ機能が実質的に存在する）
   - 中: 既存実装を流用・拡張すれば対応できる
   - 低: 類似実装なし、新規実装が必要

#### ユーザーへの確認フロー

類似実装が見つかった場合、Research深掘りの前に確認を求める。確認なしで深掘りに進まない。

**高類似度の場合:**
> 「ご依頼の機能に類似した実装が見つかりました。
>
> - `path/to/file.ts:42` — FunctionName: [一行説明]
> - `path/to/other.ts:15` — ClassName: [一行説明]
>
> これらで要件を満たせる可能性があります。以下から方針を選んでください。
>
> A. 既存実装をそのまま使う（新規実装なし）
> B. 既存実装を流用・拡張して実装する
> C. 既存実装は参考にとどめ、新規実装する」

**中類似度の場合:**
> 「ご依頼に関連する実装が見つかりました。
>
> - `path/to/file.ts:42` — FunctionName: [一行説明]
>
> この実装を拡張するアプローチが有効かもしれません。
>
> A. 既存実装を拡張する方向で計画する
> B. 既存実装は参考にとどめ、新規実装する」

**低類似度（類似なし）の場合（確認不要、そのまま進む）:**
> 「類似実装は見つかりませんでした。コードベースの詳細調査に進みます。」

ユーザーから方針A（既存をそのまま使う）を選択された場合、Phase 2以降は省略してHandoffに進む。

### 調査の進め方

対象のフォルダ・モジュールを詳細に読み、仕組み・機能・すべての詳細を理解する。表面的な読解では不十分。理解が終わったら、学習内容と発見事項の詳細なレポートをresearch.mdに書く。

ユーザーが調査対象を指定していない場合は、先に確認する:

> 「どのフォルダ/モジュールを重点的に調査しますか？ 具体的なパスがあれば教えてください。」

調査時のチェック項目:

- 関数シグネチャだけでなく、呼び出し元・呼び出し先を追う
- エラーハンドリングやエッジケースの扱いを確認する
- 既存のキャッシュ層、ミドルウェア、共通パターンを把握する
- 暗黙の前提や規約（ORM規約、命名規約、ディレクトリ構造の慣習）を特定する

### research.mdの構成

プロジェクトルートの `.claude/docs/research/` に `YYYY-MM-DD-<topic>-research.md` として保存する。

```markdown
# [調査対象] 調査レポート

日時: YYYY-MM-DD

## 対象範囲
- 調査したファイル・ディレクトリのリスト

## アーキテクチャ概要
- システム構成、データフロー、主要コンポーネントの関係

## 既存類似実装
- スキャンで見つかった類似コード（ファイルパス・関数名・概要）
- 採用方針（そのまま使う／拡張する／参考のみ／新規実装）
- 採用しない場合はその理由

## 既存パターンと規約
- 命名規約、ディレクトリ構造、エラーハンドリングのパターン
- 使用しているライブラリとその活用方法

## 重要な発見
- 既存のキャッシュ層、共通ユーティリティ、ミドルウェア
- 既に実装されている類似機能
- テスト戦略とカバレッジ

## 注意点・リスク
- 暗黙の前提や依存関係
- パフォーマンス上の制約
- 変更時に影響を受ける箇所

## バグ調査の場合（該当時のみ）
- 再現手順
- 疑わしい箇所と根拠
- 関連するログ・エラーメッセージ
```

### Research完了の基準

research.mdをユーザーに提示し、以下を確認する:

> 「research.mdを書きました。内容を確認してください。理解に誤りや不足があれば指摘をお願いします。問題なければ計画フェーズに進みます。」

ユーザーの承認を得るまでPhase 2に進まない。

## Phase 2: Plan

目的: 実装アプローチを具体的に設計し、コードスニペット付きの計画文書を作る。

### 計画の進め方

plan.mdを書く前に、変更対象となるソースファイルを実際に読む。読んでいないファイルについて変更を提案しない。実際のコードベースに基づいて計画を立てる。推測で書かない。

計画文書の要件:

- 変更対象のファイルパスを明記する
- 主要な変更内容をコードスニペットで示す
- トレードオフや代替案を記載する
- 影響範囲（既存テスト、API、他モジュール）を明記する

### 参照実装の活用

類似機能がOSSや同一コードベース内にある場合、参照として取り込む。
ゼロから設計するよりも、実績のある実装を土台にしたほうが品質が上がる。

ユーザーが参照実装を提示した場合は、それを計画に織り込む:

> 「提示いただいた実装を参考に、当プロジェクトの規約に合わせたアプローチをplan.mdに記載します。」

### plan.mdの構成

プロジェクトルートの `.claude/docs/plans/` に `YYYY-MM-DD-<feature-name>-plan.md` として保存する。

```markdown
# [機能名] 実装計画

日時: YYYY-MM-DD
関連調査: [research.mdへのパス]

## 概要
- 何を実装するか（1〜2文）
- なぜこのアプローチか

## アプローチ
- 設計方針の説明
- 代替案とそれを採用しなかった理由

## 変更内容

### [変更1: コンポーネント名]
対象: `path/to/file.ext`

（コードスニペット）

### [変更2: コンポーネント名]
対象: `path/to/file.ext`

（コードスニペット）

## 影響範囲
- 既存テストへの影響
- API変更の有無
- マイグレーションの有無

## 考慮事項
- パフォーマンス上の注意点
- セキュリティ上の注意点
- 後方互換性
```

### Plan完了の基準

plan.mdをユーザーに提示し、注釈サイクルへの移行を促す:

> 「plan.mdを書きました。内容を確認し、修正・追加したい箇所にインラインで注釈を入れてください。注釈を反映してplan.mdを更新します。」

## Phase 3: Annotation Cycle

目的: ユーザーの判断・ドメイン知識をplan.mdに注入し、計画の精度を上げる。

### 注釈サイクルの流れ

```
ユーザーがplan.mdに注釈を追加
  → Claudeが注釈を反映してplan.mdを更新
  → ユーザーが再確認
  → （1〜6回繰り返す）
  → ユーザーが承認
```

### ユーザーへの案内

注釈の入れ方をユーザーに説明する:

> plan.mdを開いて、修正したい箇所に直接メモを書き込んでください。例:
>
> - 「ここはPATCHではなくPUTを使う」
> - 「このセクション不要、削除」
> - 「drizzle:generateでマイグレーション生成、生SQLは使わない」
> - 「このフィールドはリストではなくアイテムに持たせる。スキーマセクションを修正」
>
> 注釈を入れたら教えてください。内容を反映してplan.mdを更新します。

### 注釈反映時の原則

- 注釈の意図を正確に汲み取る。不明な場合は確認する
- 注釈に従って計画を修正し、コードスニペットも更新する
- 反映後に「まだ実装しません」の姿勢を維持する
- 反映完了後、再確認を促す:

> 「注釈を反映しました。再度確認をお願いします。追加の修正があればまた注釈を入れてください。問題なければタスクリストを作成します。」

### 注釈サイクルの完了条件

ユーザーが「問題ない」「OK」「進めて」等の承認を出した時点で完了。
承認がない限り、実装に進まない。

## Phase 4: Todo List

ユーザーの承認後、計画を完了するために必要なすべてのフェーズと個々のタスクを含む詳細なToDoリストをplan.mdに追加する。まだ実装しない。

フェーズ（大区分）とタスク（個別ステップ）の2階層で構成する。各タスクは2〜5分で完了できる単位に分割する。

```markdown
## タスクリスト

### Phase 1: [フェーズ名]
- [ ] 1-1: [具体的な内容]
- [ ] 1-2: [具体的な内容]
- [ ] 1-3: [具体的な内容]

### Phase 2: [フェーズ名]
- [ ] 2-1: [具体的な内容]
- [ ] 2-2: [具体的な内容]
...
```

## Phase 5: Handoff

全フェーズ完了後、以下のメッセージで実装フェーズへ引き継ぐ:

> Research → Plan → Annotateが完了しました。
>
> - 調査レポート: `.claude/docs/research/YYYY-MM-DD-<topic>-research.md`
> - 実装計画: `.claude/docs/plans/YYYY-MM-DD-<feature-name>-plan.md`
>
> plan.mdにタスクリストが入っています。以下の指示で実装を開始できます:
>
> 「plan.mdに従ってすべてを実装してください。タスクまたはフェーズが完了したら、plan.mdで完了としてマークしてください。すべてのタスクとフェーズが完了するまで停止しないでください。不要なコメントを追加しないでください。」

## 原則

- 調査も計画もファイルに書く。チャット上の口頭説明だけで済ませない
- 「don't implement yet」を徹底する。承認前の実装は禁止
- 注釈サイクルがこのスキルの核心。ユーザーの判断を計画に織り込む工程を省略しない
- 参照実装がある場合は積極的に活用する
- research.mdは計画の根拠。plan.mdから参照できるようリンクする
- 類似実装スキャンはResearch深掘りの前の必須ステップ。重複実装を防ぐために毎回実施する
